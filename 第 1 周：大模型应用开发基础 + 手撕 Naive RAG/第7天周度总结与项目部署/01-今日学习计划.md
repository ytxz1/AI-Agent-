# 今日学习计划：周度总结与项目部署

> 目标：用 1 天时间把本周的 Naive RAG API 容器化，并形成一套可复用的部署、验收、复盘方法。

## 一、今日学习目标

完成今天后，你应该能独立回答并操作下面这些问题：

1. 为什么本地能运行的项目，换一台机器可能跑不起来？
2. Docker 镜像和容器分别是什么？
3. 为什么 Dockerfile 要先复制 `requirements.txt`，再复制业务代码？
4. 为什么 `.env` 不应该打进镜像？
5. 为什么 Chroma 的 `data/chroma` 要挂载为 volume？
6. 如何用 `docker build` 构建 RAG API 镜像？
7. 如何用 `docker run` 临时启动服务？
8. 如何用 `docker compose up` 一键启动服务？
9. 如何验证容器内 FastAPI 服务真的可用？
10. 如何判断 RAG 项目在 Docker 中是否完成了端到端闭环？

## 三、今日任务拆解

### 任务 1：确认 RAG 项目可以裸机运行

在容器化之前，先保证项目本地可以运行：

```powershell
cd "第5-6天手撕 Naive RAG 系统\naive-rag-api"
python -m venv .venv
.\.venv\Scripts\Activate.ps1
pip install -r requirements.txt
uvicorn app.main:app --reload
```

访问：

```text
http://127.0.0.1:8000/docs
```

本阶段验收：

1. 服务能启动。
2. `/health` 返回正常。
3. Swagger UI 能打开。
4. 没有 `ModuleNotFoundError`。
5. 没有 API key 读取失败。

如果裸机都不能运行，先不要急着写 Dockerfile。Docker 不是魔法，它只是把你的运行环境固化下来；项目本身坏了，放进容器还是会坏。

### 任务 2：整理环境变量

RAG 项目最少需要这些配置：

```text
APP_NAME=Naive RAG API
APP_ENV=dev
UPLOAD_DIR=data/uploads
CHROMA_DIR=data/chroma
METADATA_DIR=data/metadata
CHROMA_COLLECTION=naive_rag
CHUNK_SIZE=1000
CHUNK_OVERLAP=200
DEFAULT_TOP_K=4
CHAT_MODEL=gpt-4o-mini
EMBEDDING_MODEL=text-embedding-3-small
OPENAI_API_KEY=your_api_key_here
```

要求：

1. `.env.example` 可以提交到仓库。
2. `.env` 不提交、不打包进镜像。
3. Docker 运行时通过 `env_file` 或 `--env-file` 注入。

### 任务 3：编写 `.dockerignore`

目标：缩小构建上下文，避免把虚拟环境、缓存、密钥、向量库数据打进镜像。

必须忽略：

1. `.venv`
2. `__pycache__`
3. `.env`
4. `data`
5. `.pytest_cache`
6. `.git`

### 任务 4：编写 `Dockerfile`

核心目标：

1. 使用稳定的 Python 基础镜像。
2. 安装项目依赖。
3. 复制代码。
4. 创建必要数据目录。
5. 暴露 8000 端口。
6. 用 `uvicorn app.main:app` 启动。

### 任务 5：构建镜像

```powershell
docker build -t naive-rag-api:dev .
```

验收：

1. 构建过程没有依赖安装失败。
2. 最终出现 `naive-rag-api:dev`。
3. `docker images` 能看到镜像。

### 任务 6：用 `docker run` 验证最小运行

```powershell
docker run --rm `
  --name naive-rag-api `
  --env-file .env `
  -p 8000:8000 `
  -v "${PWD}\data:/app/data" `
  naive-rag-api:dev
```

验收：

1. 容器日志显示 Uvicorn 启动。
2. 浏览器打开 `http://127.0.0.1:8000/docs`。
3. `curl http://127.0.0.1:8000/health` 返回正常。

### 任务 7：编写 `docker-compose.yml`

目标：把长长的 `docker run` 命令沉淀为可复用配置。

启动：

```powershell
docker compose up --build
```

后台启动：

```powershell
docker compose up --build -d
```

停止：

```powershell
docker compose down
```

### 任务 8：完成 API 级验收

必须跑通四个接口：

1. `GET /health`
2. `POST /api/v1/documents/upload`
3. `GET /api/v1/documents`
4. `POST /api/v1/chat/query`

最重要的是第 4 个接口。因为 Docker 只证明服务跑起来了，RAG 问答成功才证明“加载、切分、向量化、存储、检索、生成”整条链路在容器里跑通了。

### 任务 9：验证数据持久化

步骤：

1. 启动容器。
2. 上传文档。
3. 提问成功。
4. 停止容器。
5. 重新启动容器。
6. 不重新上传文档，直接提问。

如果重启后仍然能查询到之前的文档，说明 `data/chroma` 和 `data/metadata` 的持久化基本成功。

### 任务 10：写周度总结

总结不是形式主义。它的目的有三个：

1. 把“我照着跑通了”变成“我知道为什么这样做”。
2. 把踩坑沉淀成可复用的排障清单。
3. 给下周的 Advanced RAG 学习定方向。

## 四、今日完成标准

最低标准：

1. Docker 镜像能构建。
2. 容器能启动。
3. `/health` 正常。
4. Swagger UI 正常。
5. 可以上传 `.md` 或 `.txt` 文档。
6. 可以向 RAG API 提问。

合格标准：

1. 使用 `docker compose up --build` 一键启动。
2. `.env` 不进入镜像。
3. `data` 通过 volume 挂载。
4. 容器重启后向量库数据仍然存在。
5. README 或部署文档中写清楚运行命令。

优秀标准：

1. 有健康检查。
2. 有清晰的排障清单。
3. 有 smoke test 命令。
4. 有项目复盘。
5. 能解释 Docker 部署中代码、配置、数据三者的边界。
