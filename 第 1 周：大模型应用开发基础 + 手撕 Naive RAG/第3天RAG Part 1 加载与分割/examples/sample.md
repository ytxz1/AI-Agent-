# Naive RAG 学习笔记

这是一份用于演示 RAG 文档加载与文本分块的示例 Markdown。它包含标题、表格、代码块和普通段落，方便观察不同结构在预处理过程中的表现。

## 1. RAG 是什么

RAG 的全称是 Retrieval-Augmented Generation，也就是检索增强生成。

它的核心思想是：不要只依赖大模型参数中的记忆，而是在回答问题之前，先从外部知识库中检索和问题相关的内容，再把这些内容作为上下文交给大模型。

## 2. 文档加载

文档加载的目标是把 PDF、Markdown、HTML、DOCX 等原始文件转换成统一的 Document 对象。

一个 Document 通常包含两部分：

| 字段 | 说明 |
|---|---|
| page_content | 真正参与 embedding 和检索的文本 |
| metadata | 文件来源、页码、标题路径、解析器等辅助信息 |

### 2.1 Markdown 加载

Markdown 通常有天然的标题结构，适合按标题层级切分。

例如 `#` 表示一级标题，`##` 表示二级标题，`###` 表示三级标题。

### 2.2 PDF 加载

PDF 更接近版面格式，不是天然语义格式。解析 PDF 时要检查阅读顺序、页眉页脚、表格、公式、图片说明和页码。

复杂 PDF 可以先用 MinerU 或 Docling 转换成 Markdown 或 JSON，再进入分块流程。

## 3. 文本分块

文本分块的目标是把长文档切成适合检索的知识单元。

chunk 太大时，语义会变得模糊；chunk 太小时，上下文会不完整。

### 3.1 递归字符分块

递归字符分块会优先按段落切，再按换行、句号、逗号和空格切，最后才按字符切。

```python
chain = loader | splitter | embedding_model | vector_store
```

代码块应该整体保留，不要从中间切断。

### 3.2 标题层级分块

Markdown 文档优先按标题层级切分。每个 chunk 的 metadata 中应该保留 heading_path，例如：

```text
Naive RAG 学习笔记 > 3. 文本分块 > 3.2 标题层级分块
```

这样检索结果既能回答问题，也能追踪来源。

## 4. 今日结论

RAG 的检索质量，首先取决于文档是否被正确解析、清洗、结构化和分块。
