# 第10-11天 RAG 评估体系学习计划

## 一、总目标

这两天的核心不是“知道几个指标名字”，而是形成一套工程可落地的评估闭环：

```mermaid
flowchart LR
    A["定义业务问题"] --> B["构造评估集"]
    B --> C["运行 baseline RAG"]
    C --> D["计算指标"]
    D --> E["失败样本归因"]
    E --> F["提出优化策略"]
    F --> G["运行 optimized RAG"]
    G --> H["前后对比报告"]
    H --> I["沉淀回归评估集"]
```

最终你应该能回答四个问题：

1. 我的 RAG 系统现在到底好不好？
2. 不好是因为没检索到、检索顺序差、上下文有噪声、答案不忠实，还是回答没有命中用户意图？
3. 我做的优化真的有效吗，还是只是在少数样本上看起来更好？
4. 这套评估能不能变成后续每次改 RAG 管线时的回归测试？

## 二、工具定位

| 工具 | 主要定位 | 适合今天怎么用 |
|---|---|---|
| RAGAs | 面向 RAG/LLM 应用的评估、实验、数据集、指标框架 | 主工具，用它评估 baseline 和 optimized RAG |
| FlashRAG | 面向 RAG 研究复现的 Python toolkit，包含大量 RAG 数据集、算法、组件 | 用来理解标准 RAG pipeline、检索器、重排器、生成器、压缩器和 benchmark 思维 |
| DeepEval | 类似 Pytest 的 LLM 应用测试框架，覆盖 RAG、Agent、多轮等指标 | 用来对比“测试工程化”和 RAG 指标组织方式 |
| LightEval | Hugging Face 的 LLM benchmark 工具，支持大量任务和后端 | 用来区分“模型能力评测”和“RAG 应用评测” |

### 选择原则

如果你在评估一个具体 RAG 应用，优先 RAGAs 或 DeepEval。

如果你在比较多个 RAG 研究方法、检索器、重排器、数据集，重点看 FlashRAG。

如果你在评估基础模型在 MMLU、GSM8K、TruthfulQA、IFEval 等任务上的能力，重点看 LightEval。

## 三、第10天：RAG 评估指标与评估集

### 09:00-09:30 预热：建立评估分层

先把 RAG 拆成四层：

1. Query 层：用户问题是否被正确理解、改写、扩展。
2. Retrieval 层：是否找到了正确文档，排序是否合理。
3. Context 层：送进 LLM 的上下文是否相关、完整、低噪声。
4. Generation 层：答案是否忠实、相关、完整、可读、符合格式。

记录一个口诀：

> Recall 看“有没有找全”，Precision 看“前面是不是准”，Faithfulness 看“有没有编”，Relevancy 看“有没有答到点上”。

### 09:30-10:30 指标地图

把指标按问题归类：

| 评估问题 | 指标 | 输入字段 | 优先级 |
|---|---|---|---|
| 检索结果是否把答案材料找到了？ | Context Recall、Hit Rate、Recall@K | question、retrieved_contexts、reference 或 reference_contexts | 高 |
| 相关材料是否排在前面？ | Context Precision、MRR、nDCG | question、retrieved_contexts、reference/reference_contexts | 高 |
| 上下文是否包含无关噪声？ | Context Relevancy、Noise Sensitivity | question、response、reference、retrieved_contexts | 中高 |
| 答案是否基于上下文？ | Faithfulness、Groundedness | response、retrieved_contexts | 高 |
| 答案是否回答了问题？ | Answer Relevancy | question、response | 高 |
| 答案是否和标准答案一致？ | Answer Correctness、Factual Correctness | question、response、reference | 高 |
| 端到端是否满足业务？ | Task Success、Pass/Fail Rubric、人工评分 | question、response、business rubric | 高 |
| 系统是否可上线？ | latency、cost、token usage、failure rate | traces/logs | 高 |

### 10:30-11:30 深入检索指标

重点理解：

- Hit Rate@K：前 K 个结果里是否出现至少一个相关文档。
- Recall@K：所有应该被找回的相关文档中，有多少出现在前 K 个里。
- Precision@K：前 K 个结果里有多少是相关文档。
- MRR：第一个相关文档越靠前越好。
- nDCG：适合相关性不是二值，而是 0/1/2/3 等多级评分的场景。
- RAGAs Context Precision：关注相关 chunk 是否排在更前。
- RAGAs Context Recall：关注 reference 中的关键信息是否能被 retrieved contexts 支持。

练习：拿 5 个问题手工标注 reference_context_ids，模拟计算 Hit Rate@3、Recall@5、MRR。

### 11:30-12:00 小结

输出一张“检索指标怎么选”的小表：

| 场景 | 推荐指标 |
|---|---|
| 只关心是否至少找到一个答案来源 | Hit Rate@K |
| 需要找全多个证据 | Recall@K、Context Recall |
| 需要把最有用证据排前面 | MRR、nDCG、Context Precision |
| 有文档 ID 标注 | IDBasedContextPrecision、IDBasedContextRecall |
| 没有文档 ID，只有参考答案 | LLM-based Context Precision/Recall |

### 14:00-15:30 深入生成指标

重点理解：

- Faithfulness：答案中的声明是否能被 retrieved contexts 支持。
- Answer Relevancy：答案是否对准用户问题，而不是答非所问或啰嗦偏题。
- Answer Correctness：答案和 reference 是否事实一致。
- Noise Sensitivity：系统是否被无关或有冲突的检索内容带偏。
- Format/Style/Instruction Adherence：答案是否符合约定格式、语气、长度、结构。

练习：构造 6 条坏样本：

1. 检索到了正确文档，但答案幻觉。
2. 检索到了错误文档，答案诚实地说不知道。
3. 检索到了正确文档，但答案没回答完整。
4. 检索结果正确但排序差，第一段是噪声。
5. 答案相关但不忠实。
6. 答案忠实但不相关。

### 15:30-17:00 评估集设计

评估集要覆盖真实问题分布，而不是只挑容易样本。

建议至少包含：

| 样本类型 | 占比 | 示例 |
|---|---:|---|
| 单跳事实问答 | 30% | “某个 API 的默认参数是什么？” |
| 多跳组合问答 | 20% | “A 和 B 的区别是什么，什么时候选 A？” |
| 术语解释 | 15% | “什么是 hybrid search？” |
| 操作步骤 | 15% | “如何配置 Milvus collection？” |
| 边界/无答案问题 | 10% | “文档里没有的信息，系统是否拒答？” |
| 对抗/噪声问题 | 10% | “带有错误前提或诱导性表述的问题” |

每条样本建议字段：

```json
{
  "id": "rag_eval_001",
  "question": "用户问题",
  "reference": "标准答案",
  "reference_contexts": ["支持标准答案的原文片段"],
  "reference_context_ids": ["doc_001#chunk_03"],
  "question_type": "single_hop | multi_hop | no_answer | comparison",
  "difficulty": "easy | medium | hard",
  "must_have_facts": ["必须命中的事实点"],
  "forbidden_claims": ["不允许出现的错误说法"]
}
```

## 四、第11天：用 RAGAs 跑优化前后对比

### 09:00-09:30 明确实验变量

今天只做“可解释优化”，一次只改一个关键变量。

推荐从这些优化里选 1-2 个：

| 优化项 | 预期提升 | 可能副作用 |
|---|---|---|
| top_k 从 3 调到 5/8 | Context Recall 提升 | 噪声变多，Faithfulness 可能下降 |
| 加 reranker | Context Precision 提升 | 延迟和成本增加 |
| hybrid search | Recall 和覆盖率提升 | 参数调节复杂 |
| query rewrite | 召回长尾问题 | 可能引入错误意图 |
| chunk size/overlap 调整 | 更完整证据 | chunk 太大可能稀释相关性 |
| context compression | 降噪、省 token | 可能丢失关键事实 |
| prompt 加引用约束 | Faithfulness 提升 | 答案可能更保守 |

### 09:30-10:30 搭建 RAGAs 评估脚本

你需要让 RAG 系统输出这几类字段：

```python
{
    "user_input": question,
    "response": answer,
    "retrieved_contexts": [chunk_1, chunk_2, chunk_3],
    "reference": reference_answer,
    "reference_contexts": [gold_chunk_1, gold_chunk_2],
    "retrieved_context_ids": ["doc_1#c2", "doc_3#c1"],
    "reference_context_ids": ["doc_1#c2"]
}
```

最小可用指标组合：

```python
metrics = [
    ContextPrecision(llm=evaluator_llm),
    ContextRecall(llm=evaluator_llm),
    Faithfulness(llm=evaluator_llm),
    AnswerRelevancy(llm=evaluator_llm, embeddings=evaluator_embeddings),
    NoiseSensitivity(llm=evaluator_llm),
]
```

### 14:00-15:00 失败样本归因

失败样本不要只看平均分，要逐条归因。

| 症状 | 可能原因 | 优先修复 |
|---|---|---|
| Context Recall 低 | query 没改写、embedding 不适配、chunk 切碎、top_k 太小 | query rewrite、hybrid、调 top_k、优化 chunk |
| Context Precision 低 | 召回噪声多、无 reranker、chunk 太泛 | reranker、metadata filter、chunk 重切 |
| Faithfulness 低 | prompt 没约束、模型自由发挥、上下文冲突 | 引用约束、拒答规则、冲突检测 |
| Answer Relevancy 低 | 问题理解错、答案过泛、prompt 没要求直答 | prompt 优化、query classification |
| Noise Sensitivity 高 | top_k 太大、噪声靠前、无上下文筛选 | reranker、context compression |
| Correctness 低但 Faithfulness 高 | 检索材料本身错或过期 | 更新知识库、加版本/时间过滤 |

### 15:00-16:00 实施优化

建议第一轮优化路径：

1. 如果 Context Recall 低：先加 query rewrite 或 hybrid search。
2. 如果 Context Precision 低：先加 reranker。
3. 如果 Faithfulness 低：先改 prompt，要求“只基于上下文回答；不足则说明无法从资料确认；给出引用”。
4. 如果 Noise Sensitivity 高：减少 top_k 或加 context compression/reranker。

## 五、验收标准

完成这两天后，你应该具备以下能力：

1. 能解释每个核心指标在诊断 RAG 管线中的作用。
2. 能设计一份包含真实问题、参考答案、参考上下文、难度标签的评估集。
3. 能用 RAGAs 对一批样本跑评估并保存结果。
4. 能通过指标和失败样本定位瓶颈。
5. 能做 baseline vs optimized 的可复现实验对比。
6. 能把评估流程沉淀为后续 RAG 系统迭代的回归测试。

## 六、常见误区

1. 只看平均分，不看失败样本。
2. 用 LLM Judge 分数替代人工抽检。
3. 每轮同时改很多变量，导致不知道哪个改动有效。
4. 只评估答案，不评估检索。
5. 评估集太小、太容易、太像 prompt 示例。
6. 不记录模型版本、prompt 版本、top_k、chunk 参数、索引版本。
7. 忽略成本、延迟和稳定性。

## 七、建议的复盘问题

1. 哪类问题最容易失败？
2. 失败主要发生在 retrieval 还是 generation？
3. 是否存在“指标提升但用户体验下降”的情况？
4. 优化后有没有引入新的副作用？
5. 哪些样本应该加入长期回归集？
6. 哪些指标可以自动化，哪些必须人工抽检？
