# RAG 评估数据集与实验报告模板

这份文档用于把 RAG 评估从“看概念”推进到“可执行实验”。你可以直接复制里面的字段、表格和报告结构，用于自己的 RAG 项目。

## 一、评估集设计原则

### 1. 评估集不是训练集

评估集的目标不是让系统背答案，而是稳定衡量系统变化。每次你改 chunk、embedding、top_k、reranker、prompt、模型，都用同一套评估集重新跑一遍。

### 2. 评估集要覆盖真实问题分布

不要只放简单事实问答。真实 RAG 系统的难点通常来自：

- 问题表达不清。
- 用户问题需要多段上下文组合。
- 文档中没有答案。
- 文档内容过期或冲突。
- 检索到了相似但错误的内容。
- 用户要求格式化输出。

### 3. 每条样本都应该可解释

一条高质量评估样本至少要回答：

```text
这个问题在考什么？
标准答案是什么？
答案依据在哪些文档或 chunk 中？
如果系统答错，可能是哪一类错误？
这个样本属于哪种问题类型？
```

## 二、推荐字段

### 1. 必填字段

| 字段 | 类型 | 说明 |
|---|---|---|
| id | string | 样本唯一 ID |
| question | string | 用户问题 |
| reference | string | 标准答案 |
| question_type | string | 问题类型 |
| difficulty | string | 难度 |

### 2. 强烈建议字段

| 字段 | 类型 | 说明 |
|---|---|---|
| reference_contexts | list[string] | 支持标准答案的原文片段 |
| reference_context_ids | list[string] | gold chunk ID |
| must_have_facts | list[string] | 答案必须覆盖的事实点 |
| forbidden_claims | list[string] | 答案不能出现的错误说法 |
| expected_behavior | string | answer / refuse / clarify / compare / summarize |
| tags | list[string] | 主题标签 |

### 3. 实验输出字段

| 字段 | 类型 | 说明 |
|---|---|---|
| response | string | RAG 系统答案 |
| retrieved_contexts | list[string] | 实际检索到的 chunk 文本 |
| retrieved_context_ids | list[string] | 实际检索到的 chunk ID |
| retriever_scores | list[float] | 检索分数 |
| reranker_scores | list[float] | 重排分数 |
| latency_ms | int | 端到端耗时 |
| input_tokens | int | 输入 token |
| output_tokens | int | 输出 token |
| cost_usd | float | 估算成本 |

### 4. 评估输出字段

| 字段 | 类型 | 说明 |
|---|---|---|
| context_precision | float | 相关上下文是否靠前 |
| context_recall | float | 必要证据是否找全 |
| faithfulness | float | 答案是否忠实于上下文 |
| answer_relevancy | float | 答案是否回应问题 |
| answer_correctness | float | 答案是否符合参考答案 |
| noise_sensitivity | float | 是否被噪声带偏 |
| pass | bool | 是否通过业务验收 |
| root_cause | string | 失败根因 |
| reviewer_note | string | 人工备注 |

## 三、问题类型设计

| 类型 | 说明 | 示例 |
|---|---|---|
| definition | 术语解释 | “什么是 RAGAs？” |
| single_hop | 单跳事实 | “默认 top_k 是多少？” |
| multi_hop | 多跳综合 | “为什么加 reranker 后 faithfulness 可能提升？” |
| comparison | 对比选择 | “RAGAs 和 DeepEval 有什么区别？” |
| procedure | 操作步骤 | “如何跑一次 RAGAs baseline 评估？” |
| no_answer | 文档无答案 | “这个系统支持某个未记录功能吗？” |
| adversarial | 对抗问题 | “既然文档说支持 PDF，为什么还不能导出？” |
| temporal | 时间敏感 | “当前版本是否还支持某个旧 API？” |
| citation | 引用要求 | “回答并给出对应来源。” |
| format_following | 格式约束 | “用 JSON 输出答案。” |

## 四、难度分层

| 难度 | 特征 |
|---|---|
| easy | 单个 chunk 即可回答，问题表达清晰 |
| medium | 需要 2-3 个 chunk，或需要轻微推理 |
| hard | 多跳、冲突、无答案、长上下文、问题有歧义 |

评分标准：

| 分数 | 含义 |
|---:|---|
| 0 | 不可接受，存在关键错误或严重缺失 |
| 1 | 部分可用，但需要人工修正 |
| 2 | 可直接使用 |

Pass 规则建议：

```text
correctness >= 2
faithfulness >= 2
relevance >= 2
completeness >= 1
无严重 forbidden_claim
```

## 十、失败根因标签

| 标签 | 含义 | 修复方向 |
|---|---|---|
| retrieval_miss | 没检索到必要证据 | query rewrite、hybrid、top_k、embedding、chunk |
| bad_ranking | 找到了但排太后 | reranker、metadata filter、score fusion |
| context_noise | 上下文噪声多 | reranker、compression、降低 top_n |
| chunk_boundary_issue | 答案跨 chunk 或 chunk 切坏 | 调 chunk size/overlap、语义切分 |
| stale_context | 知识过期 | 更新索引、加时间过滤 |
| conflicting_context | 文档冲突 | 冲突检测、版本优先级 |
| hallucination | 答案编造 | grounded prompt、降低温度、引用约束 |
| incomplete_answer | 答案不完整 | prompt 明确 must-have、改上下文组合 |
| irrelevant_answer | 答非所问 | query classification、prompt 优化 |
| no_answer_error | 无答案时胡编或错误拒答 | no-answer 样本、拒答策略 |
| bad_reference | 标准答案或 gold context 错 | 修评估集 |
| judge_error | 评估器误判 | 人工复核、换 evaluator、改 rubric |

## 十二、今日实际执行顺序

如果你今天要在自己的项目中马上跑起来，按这个顺序做：

1. 先写 10 条 JSONL 样本，不追求完美。
2. 确保 RAG pipeline 返回 `response` 和 `retrieved_contexts`。
3. 跑 baseline，保存 CSV。
4. 人工看 5 条高低分样本，确认指标是否合理。
5. 做一个小优化：reranker、top_k、prompt 三选一。
6. 跑 optimized，保存 CSV。
7. 填实验报告。
8. 把失败样本补充进长期评估集。

## 十三、质量门禁建议

早期项目不要把阈值设得太理想，先建立趋势。

| 阶段 | 建议门禁 |
|---|---|
| demo | pass_rate >= 60%，无严重幻觉 |
| internal beta | pass_rate >= 75%，faithfulness >= 0.75 |
| production | pass_rate >= 85%，faithfulness >= 0.85，p95 latency 达标 |
| high-stakes | 必须人工审核，自动指标只能辅助 |

## 十四、复盘问题

1. 哪个指标最能解释用户体验变化？
2. 哪些样本被优化后修复？
3. 哪些样本被优化后反而变差？
4. 变差是否来自延迟、拒答、答案过短、还是检索变化？
5. 哪个根因最值得投入下一轮优化？
6. 评估集是否需要新增 no-answer、多跳、对抗样本？
7. 这套评估能否进入 CI 或每日离线任务？
