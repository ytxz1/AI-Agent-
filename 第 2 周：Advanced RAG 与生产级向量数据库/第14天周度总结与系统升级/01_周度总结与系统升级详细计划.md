第 2 周的几个主题并不是分散知识点，而是围绕“让 RAG 从 demo 走向生产”逐步展开。

## 2. 第一周基础 RAG 的典型问题

### 2.1 数据侧问题

第一周的系统通常能跑通，但数据处理很粗：

1. 文档可能只是简单读取 txt、md、pdf 文本。
2. PDF 表格、图片、公式、页码、标题层级可能丢失。
3. chunk 只按固定长度切分，不理解标题、段落、表格、caption。
4. metadata 不完整，无法知道答案来自哪个文件、哪一页、哪个段落。
5. 没有文档版本、chunk_id、doc_id，后续无法增量更新。

升级方向：

1. 增加结构化解析。
2. 保留 `source`、`doc_id`、`chunk_id`、`page`、`section_path`。
3. 区分 text chunk、table chunk、image chunk。
4. 建立稳定的 chunk schema。

### 2.2 检索侧问题

基础 RAG 常见检索链路：

```text
query -> embedding -> vector top_k
```

问题：

1. 只依赖向量检索，容易漏掉精确术语、参数名、函数名、错误码。
2. query 太短时 embedding 表达不稳定。
3. query 包含多个问题时，单次检索覆盖不全。
4. top_k 结果不一定排序好，相关 chunk 可能排在后面。
5. 没有稀疏检索和稠密检索的互补。

升级方向：

1. 增加 BM25。
2. 增加 Milvus dense retrieval。
3. 增加 RRF 融合。
4. 增加 Reranker。
5. 增加 Query Transformation。

### 2.3 生成侧问题

常见问题：

1. 上下文拼接太粗糙，可能把无关 chunk 塞给 LLM。
2. 没有引用来源。
3. 没有要求模型只基于上下文回答。
4. 上下文过长导致成本上升。
5. 无法判断答案是否忠实于检索内容。

升级方向：

1. 只把 rerank 后的 top-n 交给 LLM。
2. Context Builder 控制 token budget。
3. Prompt 中明确要求基于上下文回答。
4. 输出 sources / citations。
5. 评估 faithfulness 与 answer relevancy。

### 2.4 工程侧问题

基础 RAG demo 常见问题：

1. 配置写死在代码里。
2. 没有日志。
3. 没有评估集。
4. 没有错误处理。
5. 没有缓存。
6. 没有模块边界。
7. 无法切换 embedding、向量库、reranker。

升级方向：

1. 配置文件化。
2. 抽象 retriever / reranker / vector store。
3. 每次 query 记录 retrieval trace。
4. 建立评估脚本。
5. 支持多策略对比。

## 3. 今日升级范围

今天不要试图一次做完所有生产能力。建议把升级范围分为三个层级。

### 3.1 必做版本

必须完成的设计或代码：

1. Milvus 替换简单向量库。
2. BM25 + Milvus 两路检索。
3. RRF 融合候选结果。
4. Reranker 对融合结果精排。
5. 输出 retrieval trace。
6. 准备 10-20 条评估 query。
7. 形成升级复盘。

### 3.2 进阶版本

有时间继续做：

1. 接入 HyDE。
2. 接入 Multi-Query。
3. 接入 Query Decomposition。
4. 支持多种 Reranker provider。
5. 建立自动化评估脚本。
6. 实现 Streamlit / Gradio 检索调试界面。

### 3.3 暂缓版本

今天先不要做太深：

1. 大规模分布式 Milvus 集群。
2. 复杂权限系统。
3. 多模态图片检索。
4. 完整在线监控系统。
5. 自动化 CI/CD。
6. 多租户隔离的完整平台化。

这些可以作为后续生产化专题。

## 5. 今日升级路线

### 阶段 1：定义 RAG v1 基线

先写清楚你现在的第一周系统是什么样。

建议记录：

```markdown
## RAG v1 当前状态

- 文档来源：
- 文档解析方式：
- chunk_size：
- chunk_overlap：
- embedding_model：
- vector_store：
- top_k：
- 是否有 BM25：
- 是否有 rerank：
- 是否有引用：
- 是否有评估：
- 最大问题：
```

输出：

- `RAG v1 架构图`
- `RAG v1 问题清单`
- `RAG v1 baseline 指标`

### 阶段 2：定义 RAG v2 目标

RAG v2 不只是“接入 Milvus”，而是完整系统升级。

建议目标：

| 维度 | RAG v1 | RAG v2 |
|---|---|---|
| 数据处理 | 简单文本读取 | 结构化解析与 chunk metadata |
| 向量库 | 内存 / 本地轻量库 | Milvus |
| 检索 | 单向量检索 | BM25 + Milvus 混合检索 |
| 排序 | embedding score | Reranker 二阶段精排 |
| Query | 原始问题 | HyDE / Multi-Query / Decomposition 可选 |
| 生成 | 简单拼上下文 | Context Builder + citations |
| 日志 | 无 | retrieval_trace |
| 评估 | 人工感觉 | 指标化评估 |
| 配置 | 写死 | yaml / env |

### 阶段 3：升级数据入库链路

从：

```text
documents -> chunks -> embeddings -> vector store
```

升级为：

```text
documents
  -> parser
  -> normalized blocks
  -> structure-aware chunks
  -> embeddings
  -> Milvus collection
  -> BM25 index
```

关键点：

1. chunk 必须有稳定 `chunk_id`。
2. chunk 必须保留 source metadata。
3. Milvus 中保存 vector + text + metadata。
4. BM25 使用同一批 chunk 建索引。
5. 数据版本要记录，方便评估复现。

### 阶段 4：升级查询链路

从：

```text
query -> vector search -> answer
```

升级为：

```text
query
  -> optional query transformation
  -> bm25 retrieve
  -> milvus retrieve
  -> fusion
  -> rerank
  -> context build
  -> answer
  -> trace log
```

关键点：

1. BM25 和 Milvus 可以并行。
2. 两路结果不要直接拼接，优先用 RRF。
3. Reranker 输入候选数量要足够。
4. LLM 只吃 rerank 后 top-n。
5. 每一步都记录 trace。

### 阶段 5：建立评估闭环

评估顺序：

1. 先评估检索，不急着评估最终答案。
2. 再评估 rerank 后 top-n。
3. 最后评估 answer faithfulness。

推荐对比：

| 策略 | 说明 |
|---|---|
| baseline_vector | 第一周原始向量检索 |
| milvus_dense | Milvus 稠密检索 |
| bm25_only | 只用 BM25 |
| hybrid_rrf | BM25 + Milvus + RRF |
| hybrid_rerank | BM25 + Milvus + RRF + Reranker |
| hyde_hybrid_rerank | HyDE + Hybrid + Reranker |
| multi_query_hybrid_rerank | Multi-Query + Hybrid + Reranker |

## 7. 今天最重要的判断框架

### 7.1 什么时候用 Query Transformation

| Query 类型 | 建议 |
|---|---|
| 很短、抽象 | HyDE |
| 口语化、表达不准 | Query Rewrite |
| 角度宽泛 | Multi-Query |
| 多个子问题 | Decomposition |
| 精确事实、编号、参数 | 原始 query + metadata filter |

### 7.2 什么时候用 BM25

BM25 适合：

1. 参数名。
2. 函数名。
3. 报错信息。
4. 产品名。
5. 文件名。
6. 专有名词。
7. 原文必须出现某个关键词的查询。

### 7.3 什么时候用 Milvus dense retrieval

Milvus dense retrieval 适合：

1. 语义相似问题。
2. 概念解释问题。
3. 用户表达和文档表达不一致。
4. 需要跨段落找到相关说明。
5. 自然语言问答。

### 7.4 什么时候用 Reranker

Reranker 适合：

1. 召回候选多，但噪声也多。
2. top-3 质量影响答案明显。
3. LLM 上下文窗口有限。
4. 用户问题复杂，需要更细粒度相关性判断。
5. 混合检索后需要统一排序。

## 9. 建议结论

今天的核心收获应该是：生产级 RAG 不是一个单点算法，而是一条可调试、可观测、可评估的检索生成链路。

第一周的基础 RAG 解决“能不能跑通”；第二周的 Advanced RAG 解决“能不能稳定、可解释、可优化地跑好”。Milvus、Hybrid Retrieval、Reranker、Query Transformation、Evaluation 分别对应系统的不同瓶颈，只有把它们组合成闭环，RAG 才真正开始接近工程落地。
