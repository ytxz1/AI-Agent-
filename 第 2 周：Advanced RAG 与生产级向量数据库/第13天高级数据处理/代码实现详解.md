# 第13天高级数据处理代码实现详解

> 目标：用具体代码实现一个复杂 PDF 解析流水线，把 Unstructured / MinerU / Docling 的输出统一成 RAG 可用的数据结构，并生成 chunk 与质量评估报告。  
> 适用场景：包含表格、图片、扫描页、复杂版面、多栏论文、公式的 PDF。

## 3. 为什么要设计统一的 NormalizedBlock

Unstructured、MinerU、Docling 的输出形态完全不同。

Unstructured 更像这样：

```json
{
  "category": "Table",
  "text": "...",
  "metadata": {
    "page_number": 2,
    "text_as_html": "<table>...</table>"
  }
}
```

MinerU 更像这样：

```json
{
  "type": "table",
  "table_caption": ["表 1 各区域收入情况"],
  "table_body": "<table>...</table>",
  "img_path": "images/table_1.jpg",
  "page_idx": 1,
  "bbox": [90, 240, 900, 620]
}
```

Docling 可以导出 Markdown / JSON，但它的内部结构又是另一套文档模型。

如果你直接拿这三种输出去做 RAG，会遇到几个问题：

1. chunk 逻辑要写三遍。
2. 评估逻辑要写三遍。
3. 表格、图片、公式的字段名字不统一。
4. 后续换解析器时，向量库 payload 结构也会变。
5. 多解析器融合很困难。

所以代码里定义了统一结构：

```python
@dataclass
class NormalizedBlock:
    document_id: str
    source_file: str
    parser: str
    block_id: str
    block_type: str
    text: str = ""
    html: str | None = None
    image_path: str | None = None
    page_idx: int | None = None
    bbox: Any | None = None
    parent_id: str | None = None
    section_path: list[str] = field(default_factory=list)
    metadata: dict[str, Any] = field(default_factory=dict)
```

字段解释：

| 字段 | 作用 |
|---|---|
| `document_id` | 文档 ID，通常用 PDF 文件名 |
| `source_file` | 原始文件路径，用于溯源 |
| `parser` | 来源解析器，例如 `mineru`、`unstructured`、`docling` |
| `block_id` | block 唯一 ID |
| `block_type` | 标准块类型，例如 `text`、`title`、`table`、`image`、`equation` |
| `text` | 可参与 embedding 的文本 |
| `html` | 表格 HTML 或富结构内容 |
| `image_path` | 图片、表格截图、公式截图路径 |
| `page_idx` | 页码，统一建议从 0 开始 |
| `bbox` | 页面坐标，用于高亮和可视化引用 |
| `parent_id` | 父级结构，暂时保留 |
| `section_path` | 标题路径，例如 `["2 方法", "2.1 数据集"]` |
| `metadata` | 原始解析器元数据完整保留 |

这就是整个工程最重要的一层。

## 4. `scripts/common.py` 详解

`common.py` 是公共基础库，其他脚本都依赖它。

### 4.1 `NormalizedBlock`

这是解析之后的统一 block。

一个真实的表格 block 可能长这样：

```json
{
  "document_id": "demo_report",
  "source_file": "data/raw_pdfs/demo_report.pdf",
  "parser": "mineru",
  "block_id": "demo_report-mineru-2",
  "block_type": "table",
  "text": "表 1 各区域收入情况...",
  "html": "<table>...</table>",
  "image_path": "outputs/mineru/demo_report/images/table_1.jpg",
  "page_idx": 1,
  "bbox": [90, 240, 900, 620],
  "section_path": ["2025 年度经营分析报告"],
  "metadata": {
    "raw_type": "table"
  }
}
```

### 4.2 `RagChunk`

这是最终进入向量库前的 chunk。

```python
@dataclass
class RagChunk:
    chunk_id: str
    document_id: str
    source_file: str
    parser: str
    chunk_type: str
    text: str
    source_blocks: list[str]
    page_range: list[int | None]
    section_path: list[str] = field(default_factory=list)
    html: str | None = None
    image_path: str | None = None
    metadata: dict[str, Any] = field(default_factory=dict)
```

为什么 chunk 还要保存 `source_blocks`？

因为向量检索返回 chunk 后，你需要知道它来自哪些原始 block。这样才能：

- 回到原 PDF 页码。
- 展示表格 HTML。
- 展示图片截图。
- 高亮 PDF 坐标。
- 做答案引用。
- 做错误排查。

### 4.3 `iter_input_pdfs`

作用：既支持输入单个 PDF，也支持输入一个目录。

```python
def iter_input_pdfs(input_path: Path) -> list[Path]:
    if input_path.is_file() and input_path.suffix.lower() in PDF_SUFFIXES:
        return [input_path]
    if input_path.is_dir():
        return sorted(p for p in input_path.rglob("*") if p.suffix.lower() in PDF_SUFFIXES)
    raise FileNotFoundError(...)
```

所以你可以这样跑：

```powershell
python scripts/parse_unstructured.py --input data/raw_pdfs/demo.pdf --output outputs/unstructured
```

也可以这样跑：

```powershell
python scripts/parse_unstructured.py --input data/raw_pdfs --output outputs/unstructured
```

### 4.4 `normalize_text`

作用：清理文本空白。

```python
def normalize_text(text: str | None) -> str:
    if not text:
        return ""
    text = text.replace("\r\n", "\n").replace("\r", "\n")
    text = re.sub(r"[ \t]+", " ", text)
    text = re.sub(r"\n{3,}", "\n\n", text)
    return text.strip()
```

这一步很关键。PDF 解析输出经常出现：

- 多余空格。
- 多余换行。
- Windows / Unix 换行混杂。
- 空文本 block。

如果不清理，embedding 质量会下降，chunk 也会变得很碎。

## 5. Unstructured 解析代码详解

文件：

```text
scripts/parse_unstructured.py
```

### 5.1 核心函数

```python
elements = partition_pdf(**kwargs)
```

这里调用 Unstructured 的 `partition_pdf`，它会把 PDF 解析成元素列表。

脚本默认参数：

```python
kwargs = {
    "filename": str(pdf_path),
    "strategy": strategy,
    "languages": languages,
    "include_page_breaks": True,
    "infer_table_structure": infer_table_structure,
}
```

### 5.2 参数解释

`strategy=hi_res`：

- 使用更强的布局解析。
- 适合复杂 PDF。
- 速度比 `fast` 慢。
- 依赖更多模型和系统库。

`infer_table_structure=True`：

- 尝试恢复表格结构。
- 目标是拿到 `metadata.text_as_html`。
- 没有这个参数时，表格可能只是纯文本。

`languages=["eng", "chi_sim"]`：

- OCR 语言。
- 英文 + 简体中文。
- 如果 Tesseract 没装中文语言包，中文 OCR 会失败或质量差。

`include_page_breaks=True`：

- 保留分页边界。
- 方便后续页码溯源。

### 5.3 图片块抽取

当你传入：

```powershell
--extract-image-blocks
```

脚本会增加：

```python
kwargs.update(
    {
        "extract_image_block_types": ["Image", "Table"],
        "extract_image_block_to_payload": False,
        "extract_image_block_output_dir": str(image_output_dir),
    }
)
```

作用：

- 抽取图片 block。
- 抽取表格截图。
- 保存到当前文档的 `assets/` 目录。

输出结构示例：

```text
outputs/unstructured/financial_report/
├── elements.json
├── preview.md
├── stats.json
├── assets/
└── tables/
```

### 5.4 `elements.json`

脚本会把每个 element 转成 JSON：

```python
{
    "index": idx,
    "source_file": str(source_pdf),
    "category": element.category,
    "text": str(element),
    "metadata": metadata,
}
```

这样做的好处：

- 原始输出被保留。
- 标准化脚本可以离线重跑。
- 你不用每次修改 normalize 逻辑都重新解析 PDF。

## 8. 标准化脚本详解

### 8.1 Unstructured 标准化

文件：

```text
scripts/normalize_unstructured.py
```

核心映射：

```python
TYPE_MAP = {
    "Title": "title",
    "NarrativeText": "text",
    "Text": "text",
    "ListItem": "list",
    "Table": "table",
    "Image": "image",
    "PageBreak": "page_break",
    "Header": "header",
    "Footer": "footer",
}
```

页码处理：

```python
page_number = metadata.get("page_number")
page_idx = page_number - 1 if isinstance(page_number, int) else page_number
```

为什么减 1？

- Unstructured 的 `page_number` 通常从 1 开始。
- MinerU 的 `page_idx` 通常从 0 开始。
- 为了统一，内部标准使用从 0 开始的 `page_idx`。

表格 HTML：

```python
html=metadata.get("text_as_html")
```

坐标：

```python
bbox=metadata.get("coordinates")
```

### 8.2 MinerU 标准化

文件：

```text
scripts/normalize_mineru.py
```

它会自动查找：

```text
*content_list.json
*_content_list.json
```

核心字段拼接：

```python
text = first_existing_text(
    item,
    [
        "text",
        "table_caption",
        "table_body",
        "image_caption",
        "img_caption",
        "latex",
    ],
)
```

为什么要这样拼？

因为复杂 PDF 中很多 block 不只是纯文本。

表格 block 可能有：

- 表格标题。
- 表格 HTML。
- 表格截图。

图片 block 可能有：

- 图片标题。
- 图片路径。
- 图片附近文字。

公式 block 可能有：

- LaTeX。
- 公式截图。

如果只取 `text` 字段，很多重要内容会丢。

标题层级处理：

```python
text_level = item.get("text_level")
if block_type == "text" and isinstance(text_level, int):
    block_type = "title"
```

MinerU 的标题有时表现为 `type=text` 加 `text_level`。代码会把它转成标准 `title`。

图片路径处理：

```python
image_path = item.get("img_path") or item.get("image_path")
if image_path and not Path(image_path).is_absolute():
    image_path = str((content_list_path.parent / image_path).resolve())
```

为什么转绝对路径？

- 后续展示图片时不容易丢。
- 向量库 payload 里可以保存稳定路径。
- 多脚本从不同工作目录运行时也能找到资产。

### 8.3 Docling 标准化

文件：

```text
scripts/normalize_docling.py
```

当前做法：

1. 读取 `docling.md`。
2. 用 Markdown 标题识别章节。

## 10. Chunk 构建详解

文件：

```text
scripts/build_chunks.py
```

### 10.1 为什么不直接固定长度切分

复杂 PDF 不能只按 1000 字切分。

错误示例：

```text
表格前半段进入 chunk A
表格后半段进入 chunk B
caption 被切到 chunk C
图片路径丢失
```

这样会导致：

- 表格检索不到。
- 检索到了也无法展示完整证据。
- LLM 看到半张表，会编答案。

所以脚本采用结构感知 chunking。

### 10.2 普通文本聚合

普通文本类型：

```python
TEXT_TYPES = {"title", "text", "list", "header", "footer"}
```

脚本会把这些 block 聚合到不超过 `max_tokens` 的 chunk。

默认：

```powershell
--max-tokens 900
--overlap-blocks 1
```

注意这里的 token 是粗略估算：

```python
def approx_tokens(text: str) -> int:
    cjk_chars = sum(1 for ch in text if "\u4e00" <= ch <= "\u9fff")
    other_words = len([w for w in text.split() if w])
    return cjk_chars + other_words
```

为什么不用真实 tokenizer？

- 今天的目标是数据处理主线。
- 不想引入额外依赖。
- 中文场景下粗估已经足够控制 chunk 大小。

生产升级可以换成：

- `tiktoken`
- `transformers` tokenizer
- 你实际 embedding 模型对应 tokenizer

### 10.3 特殊块单独成 chunk

特殊类型：

```python
SPECIAL_TYPES = {"table", "image", "chart", "equation"}
```

这些不会和普通正文混在一个大 chunk 里，而是单独成 chunk。

原因：

- 表格要保留完整 HTML。
- 图片要保留 `image_path`。
- 公式要保留 LaTeX 或截图。
- 图表后续可能要走多模态处理。

### 10.4 chunk 输出示例

```json
{
  "chunk_id": "demo_report-mineru-chunk-0001-c6741375a1e1",
  "document_id": "demo_report",
  "source_file": "data/raw_pdfs/demo_report.pdf",
  "parser": "mineru",
  "chunk_type": "table",
  "text": "2025 年度经营分析报告\n表 1 各区域收入情况...",
  "source_blocks": ["demo_report-mineru-2"],
  "page_range": [1, 1],
  "section_path": ["2025 年度经营分析报告"],
  "html": "<table>...</table>",
  "image_path": "outputs/mineru/demo_report/images/table_1.jpg",
  "metadata": {
    "block_types": ["table"],
    "approx_tokens": 70
  }
}
```

这个 chunk 可以直接写入向量库：

- `text` 用于 embedding。
- `html` 用于表格展示。
- `image_path` 用于展示原图。
- `page_range` 用于引用页码。
- `source_blocks` 用于追溯原始 block。

## 11. 质量评估脚本详解

文件：

```text
scripts/evaluate_parse_quality.py
```

这个脚本不是 OCR 精度评估，也不是完全自动的语义质量评估。它做的是结构健康检查。

输出：

```text
outputs/reports/parse_quality_report.json
```

### 11.1 统计维度

```json
{
  "total_blocks": 4,
  "by_parser": {
    "mineru": 4
  },
  "by_type": {
    "image": 1,
    "table": 1,
    "text": 1,
    "title": 1
  }
}
```

这能帮你判断：

- 某个解析器有没有输出。
- 表格是否被识别。
- 图片是否被识别。
- 标题是否被识别。

### 11.2 文本质量指标

```json
"text": {
  "with_text_ratio": 1.0,
  "avg_text_chars": 57.5,
  "empty_text_blocks": 0
}
```

如果 `empty_text_blocks` 很高，说明：

- 解析器产生了大量空块。
- 图片 / 表格没有 caption。
- OCR 失败。
- 标准化逻辑没有取对字段。

### 11.3 元数据指标

```json
"metadata": {
  "with_page_ratio": 1.0,
  "with_bbox_ratio": 1.0,
  "with_section_path_ratio": 1.0
}
```

这些指标对生产 RAG 很关键。

如果没有 `page_idx`：

- 答案无法引用页码。

如果没有 `bbox`：

- 不能在 PDF 上高亮证据区域。

如果没有 `section_path`：

- chunk 缺少章节上下文，检索语义会变弱。

### 11.4 表格指标

```json
"tables": {
  "table_blocks": 1,
  "with_html_ratio": 1.0
}
```

如果表格很多，但 `with_html_ratio` 很低：

- Unstructured 可能没有打开 `infer_table_structure=True`。
- MinerU 可能没有正确输出 `table_body`。
- 表格可能是扫描图片，需要 OCR / 表格识别 fallback。

### 11.5 图片资产指标

```json
"assets": {
  "blocks_with_image_path": 2,
  "existing_image_path_ratio": 0.0
}
```

如果 `existing_image_path_ratio` 低：

- 图片路径可能是相对路径。
- 文件被移动。
- 标准化时路径拼接错了。
- 解析器没有真的导出图片。

## 14. 如何接入向量库

当前 `rag_chunks.jsonl` 已经接近可以直接入库。

每行一个 chunk：

```json
{
  "chunk_id": "...",
  "text": "...",
  "html": "...",
  "image_path": "...",
  "metadata": {
    "approx_tokens": 70
  }
}
```

### 14.1 写入 Milvus 的思路

字段设计：

| 字段 | 类型 | 说明 |
|---|---|---|
| `chunk_id` | VarChar | 主键 |
| `embedding` | FloatVector | 文本向量 |
| `document_id` | VarChar | 文档 ID |
| `source_file` | VarChar | 原始文件 |
| `parser` | VarChar | 解析器 |
| `chunk_type` | VarChar | text/table/image/equation |
| `text` | VarChar | chunk 文本 |
| `page_start` | Int64 | 起始页 |
| `page_end` | Int64 | 结束页 |
| `payload_json` | JSON | html、image_path、section_path 等 |

伪代码：

```python
for chunk in chunks:
    vector = embedding_model.embed_query(chunk["text"])
    row = {
        "chunk_id": chunk["chunk_id"],
        "embedding": vector,
        "document_id": chunk["document_id"],
        "source_file": chunk["source_file"],
        "parser": chunk["parser"],
        "chunk_type": chunk["chunk_type"],
        "text": chunk["text"],
        "page_start": chunk["page_range"][0],
        "page_end": chunk["page_range"][1],
        "payload_json": {
            "html": chunk["html"],
            "image_path": chunk["image_path"],
            "section_path": chunk["section_path"],
            "source_blocks": chunk["source_blocks"],
            "metadata": chunk["metadata"],
        },
    }
```

### 14.2 查询时如何展示

如果命中 `chunk_type=text`：

- 展示文本。
- 展示页码。
- 展示章节路径。

如果命中 `chunk_type=table`：

- 展示表格摘要文本。
- 展示 HTML 表格。
- 展示表格截图。
- 展示 PDF 页码。

如果命中 `chunk_type=image`：

- 展示 caption / 视觉摘要。
- 展示图片。
- 展示附近正文。

如果命中 `chunk_type=equation`：

- 展示 LaTeX。
- 展示公式截图。
- 展示前后解释段落。

## 15. 常见修改点

### 15.1 想让 chunk 更大

改：

```powershell
python scripts/build_chunks.py --max-tokens 1400
```

适合：

- 长报告。
- 段落比较完整。
- embedding 模型上下文更长。

### 15.2 想让 chunk 更小

改：

```powershell
python scripts/build_chunks.py --max-tokens 500
```

适合：

- 问答粒度很细。
- 检索需要更精确。
- 表格和图片较多。

### 15.3 想去掉页眉页脚

可以在 `build_chunks.py` 里过滤：

```python
if block_type in {"header", "footer"}:
    continue
```

也可以在标准化阶段过滤。

生产建议：

- 不要一开始直接删除。
- 先保留，在质量报告里观察。
- 对重复页眉页脚写规则后再删除。

### 15.4 想强化表格摘要

当前表格文本是：

```text
caption + table_body
```

更好的生产做法：

1. 把 HTML 转 Markdown。
2. 提取表头。
3. 抽样关键行。
4. 用 LLM 生成表格摘要。
5. embedding 摘要，但 payload 保留完整 HTML。

### 15.5 想强化图片理解

当前图片文本主要来自 caption。

更好的生产做法：

1. 读取 `image_path`。
2. 调用多模态模型生成视觉描述。
3. 对图片做 OCR。
4. 合并 caption + OCR + 视觉描述 + 附近正文。
5. 再进入 embedding。

## 18. 最重要的理解

复杂 PDF 处理不要追求“一步到位的完美解析”。

更可靠的工程思路是：

```text
保留原始解析输出
  -> 标准化成统一 block
  -> 记录解析器来源
  -> 特殊结构单独处理
  -> 生成质量报告
  -> 用错误案例持续改进
```

只要 `NormalizedBlock` 稳定，后面你可以随时替换解析器、增强 OCR、增加多模态模型、接向量库、做人工审阅，而不会把整条 RAG 管线推翻重写。
