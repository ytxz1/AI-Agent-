```text

  -> 主键 get
  -> upsert 更新
  -> delete 删除
```

## 4. 配置常量解释

代码：

```python
DEFAULT_URI = "http://localhost:19530"
DEFAULT_TOKEN = "root:Milvus"
DEFAULT_COLLECTION = "day12_rag_chunks"

```

解释：

| 常量 | 作用 |
| --- | --- |
| `DEFAULT_URI` | Milvus Standalone 默认连接地址 |
| `DEFAULT_TOKEN` | Milvus 默认账号密码 token |
| `DEFAULT_COLLECTION` | 本次 demo 使用的 collection 名称 |
| `VECTOR_DIM` | 向量维度 |

重点：

- Milvus collection 的向量维度是固定的。
- 你创建 collection 时写 `dim=128`，后续插入的每个向量都必须是 128 维。
- 如果真实 embedding 模型输出 768 维，collection 就必须建成 768 维。
- 如果真实 embedding 模型输出 1536 维，collection 就必须建成 1536 维。

## 5. DocumentChunk 数据结构

代码：

```python
@dataclass(frozen=True)
class DocumentChunk:
    id: int
    doc_id: str
    chunk_id: str
    title: str
    text: str
    source: str
    category: str
    tenant_id: str
    page: int
    created_at: int
```

这个类表示 RAG 系统中的一个文档块。

为什么不是直接存整个 PDF？

因为 RAG 检索通常不是按“整篇文档”召回，而是按 chunk 召回。

例如一篇 30 页 PDF：

```text
PDF
  -> page 1 chunk 1
  -> page 1 chunk 2
  -> page 2 chunk 1
  -> ...
```

每个 chunk 都应该有：

| 字段 | 解释 |
| --- | --- |
| `id` | Milvus 主键 |
| `doc_id` | 原始文档 ID |
| `chunk_id` | 文档块 ID |
| `title` | 标题 |
| `text` | chunk 原文 |
| `source` | 来源，例如文件名、URL、知识库名 |
| `category` | 分类，例如 milvus、rag、qdrant |
| `tenant_id` | 租户 ID，用于权限隔离 |
| `page` | 页码或位置 |
| `created_at` | 创建时间戳 |

生产中尤其重要的是：

- `tenant_id`：多租户隔离。
- `doc_id`：删除或更新整篇文档时用。
- `chunk_id`：定位原文片段时用。
- `source`：生成答案时做 citation。

## 6. 为什么使用本地伪 embedding

真实 RAG 一定要使用真正的 embedding 模型，例如：

- OpenAI embedding。
- BGE。
- E5。
- GTE。
- Jina embedding。

但是今天的目标是学习 Milvus SDK。

如果一上来就接真实模型，会多出很多干扰：

- API key。
- 网络。
- 模型下载。
- CUDA/CPU 环境。
- 代理。
- 依赖冲突。

所以脚本使用一个本地确定性 embedding：

```python
def embed_text(text: str, dim: int = VECTOR_DIM) -> list[float]:
    ...
```

它的特点：

- 不需要联网。
- 不需要下载模型。
- 同一段文本每次生成同一个向量。
- 可以练习 Milvus 的 insert/search/filter。
- 不代表真实语义能力。

你可以把它理解为“SDK 训练轮”。

## 7. tokenize()：文本切分

代码：

```python
def tokenize(text: str) -> list[str]:
    text = text.lower()
    return re.findall(r"[a-z0-9_]+|[\u4e00-\u9fff]", text)
```

作用：

- 英文按单词切。
- 数字按连续数字切。
- 中文按单字切。

示例：

```python
tokenize("Milvus Docker 部署端口 19530")
```

可能得到：

```python
["milvus", "docker", "部", "署", "端", "口", "19530"]
```

为什么中文按字切？

因为我们不想引入额外分词依赖，例如 `jieba`。这不是最好的中文检索方式，但足够用于本地 demo。

## 8. stable_hash()：稳定哈希

代码：

```python
def stable_hash(value: str) -> int:
    digest = hashlib.sha256(value.encode("utf-8")).hexdigest()
    return int(digest[:16], 16)
```

为什么不用 Python 内置 `hash()`？

因为 Python 的内置 `hash()` 默认带随机盐：

```python
hash("milvus")
```

不同进程可能结果不同。

我们需要：

```text
同一段文本 -> 同一个向量
```

所以用 `sha256`。

## 9. embed_text()：生成向量

核心代码：

```python
vector = [0.0] * dim

for token in tokens:
    hashed = stable_hash(token)
    index = hashed % dim
    sign = 1.0 if ((hashed >> 8) & 1) == 0 else -1.0
    vector[index] += sign

norm = math.sqrt(sum(value * value for value in vector))
return [value / norm for value in vector]
```

它做了几件事：

1. 创建一个 128 维零向量。
2. 对每个 token 做稳定哈希。
3. 把 token 映射到某个向量维度。
4. 在该维度上加 1 或减 1。
5. 最后做 L2 归一化。

为什么要归一化？

因为我们后面使用 `COSINE` 相似度。归一化后，向量方向更重要，长度影响更小。

再次强调：

这个函数不是生产 embedding。生产中应替换为真实模型。

## 11. chunk_to_entity()：转成 Milvus 可插入格式

代码：

```python
def chunk_to_entity(chunk: DocumentChunk) -> dict[str, Any]:
    return {
        "id": chunk.id,
        "doc_id": chunk.doc_id,
        "chunk_id": chunk.chunk_id,
        "title": chunk.title,
        "text": chunk.text,
        "source": chunk.source,
        "category": chunk.category,
        "tenant_id": chunk.tenant_id,
        "page": chunk.page,
        "created_at": chunk.created_at,
        "vector": embed_text(chunk.title + "\n" + chunk.text),
    }
```

Milvus 插入数据时，每一条记录就是一个 dict。

关键点：

- dict 的 key 必须和 schema 字段名一致。
- `vector` 字段必须是 `list[float]`。
- `vector` 长度必须等于 `VECTOR_DIM`。
- `id` 是主键，不能重复，除非使用 upsert 覆盖。

为什么 embedding 用 `title + text`？

因为标题通常含有高价值语义信息。

例如：

```text
标题：Milvus Docker 部署
正文：默认服务端口是 19530
```

比只 embedding 正文更容易被“Milvus 怎么部署”召回。

## 13. assert_milvus_available()：连接检查

代码：

```python
def assert_milvus_available(client: MilvusClient) -> None:
    try:
        client.list_collections()
    except Exception as exc:
        raise RuntimeError(...)
```

为什么用 `list_collections()` 检查？

因为它是一个轻量操作：

- 如果 Milvus 正常，会返回 collection 列表。
- 如果 Milvus 没启动，会立即报连接错误。

常见失败原因：

- Docker 没启动。
- Milvus 容器没启动。
- 端口不是 `19530`。
- WSL2 和 Windows 网络没打通。
- token 写错。

## 14. recreate_collection()：重建 Collection

这部分是整份代码的核心。

代码逻辑：

```python
if client.has_collection(collection_name):
    client.drop_collection(collection_name)

schema = MilvusClient.create_schema(...)
schema.add_field(...)
index_params = client.prepare_index_params()
index_params.add_index(...)
client.create_collection(...)
```

### 14.1 为什么先 drop

学习 demo 需要每次运行都得到同样的结果。

如果不删除旧 collection：

- 主键可能重复。
- schema 可能已经变了。
- 旧数据会影响搜索结果。

所以 demo 中直接删除再创建。

生产中不要随便 drop collection。生产应该走迁移脚本、灰度、备份、恢复流程。

### 14.2 schema 是什么

schema 就是 collection 的字段定义。

你可以把 Milvus collection 理解成关系数据库的一张表：

```text
collection: day12_rag_chunks

id          INT64 primary key
vector      FLOAT_VECTOR(128)
doc_id      VARCHAR(128)
chunk_id    VARCHAR(128)
title       VARCHAR(512)
text        VARCHAR(4096)
source      VARCHAR(512)
category    VARCHAR(128)
tenant_id   VARCHAR(128)
page        INT64
created_at  INT64
```

### 14.3 主键字段

代码：

```python
schema.add_field("id", DataType.INT64, is_primary=True)
```

含义：

- `id` 是主键。
- 每条记录必须有唯一 id。
- 后续可以用 `client.get(ids=[...])` 直接按主键查询。

### 14.4 向量字段

代码：

```python
schema.add_field("vector", DataType.FLOAT_VECTOR, dim=VECTOR_DIM)
```

含义：

- 字段名叫 `vector`。
- 类型是浮点向量。
- 维度是 128。

这是 Milvus 最关键的字段。

### 14.5 标量字段

代码：

```python
schema.add_field("category", DataType.VARCHAR, max_length=128)
schema.add_field("tenant_id", DataType.VARCHAR, max_length=128)
schema.add_field("page", DataType.INT64)
```

标量字段用于：

- 过滤。
- 权限控制。
- 来源展示。
- 文档追踪。

例如：

```python
filter='tenant_id == "team_a" and category == "rag"'
```

### 14.6 enable_dynamic_field=False

代码：

```python
schema = MilvusClient.create_schema(
    auto_id=False,
    enable_dynamic_field=False,
)
```

含义：

- `auto_id=False`：我们自己传主键 id。
- `enable_dynamic_field=False`：不允许插入 schema 外的字段。

学习阶段建议关掉 dynamic field。

原因：

- 更容易发现字段写错。
- 更接近生产 schema 管理。
- 不会悄悄把错误字段塞进动态 JSON。

## 15. 创建索引

代码：

```python
index_params = client.prepare_index_params()
index_params.add_index(
    field_name="vector",
    index_type="AUTOINDEX",
    metric_type="COSINE",
)
```

解释：

| 参数 | 含义 |
| --- | --- |
| `field_name="vector"` | 对哪个向量字段建索引 |
| `index_type="AUTOINDEX"` | 使用 Milvus 自动索引 |
| `metric_type="COSINE"` | 使用余弦相似度 |

为什么用 AUTOINDEX？

Milvus 官方文档说明，ANN 搜索依赖预建索引，而不同索引会影响吞吐、内存和召回；AUTOINDEX 可以降低学习曲线，让 Milvus 根据数据情况选择较合适的索引参数。

等你熟悉后，可以改成 HNSW：

```python
index_params.add_index(
    field_name="vector",
    index_type="HNSW",
    metric_type="COSINE",
    params={
        "M": 16,
        "efConstruction": 200,
    },
)
```

HNSW 常见参数：

| 参数 | 影响 |
| --- | --- |
| `M` | 图连接数，越大召回可能越好，内存越高 |
| `efConstruction` | 建索引质量，越大构建越慢，召回可能越好 |
| `ef` | 搜索候选数量，越大召回越好，延迟越高 |

## 17. insert_chunks()：插入数据

代码：

```python
chunks = build_sample_chunks()
entities = [chunk_to_entity(chunk) for chunk in chunks]

result = client.insert(
    collection_name=collection_name,
    data=entities,

client.flush(collection_name=collection_name)
```

解释：

2. `chunk_to_entity()` 给每个文档块生成向量。
3. `client.insert()` 写入 Milvus。
4. `client.flush()` 将数据刷入存储。

插入时最容易出错的是：

| 问题 | 原因 |
| --- | --- |
| 维度错误 | `vector` 长度不是 128 |
| 字段错误 | dict key 和 schema 不一致 |
| 字符串过长 | 超过 VARCHAR max_length |
| 主键重复 | 相同 id 已经存在 |
| 类型错误 | page 传成字符串，id 传成字符串 |

## 19. vector_search()：核心搜索函数

代码：

```python
def vector_search(
    client: MilvusClient,
    collection_name: str,
    query: str,
    limit: int = 3,
    filter_expr: str | None = None,
) -> list[Any]:
    query_vector = embed_text(query)

    return client.search(
        collection_name=collection_name,
        anns_field="vector",
        data=[query_vector],
        limit=limit,
        filter=filter_expr,
        output_fields=[...],
        search_params={"metric_type": "COSINE"},
    )
```

参数解释：

| 参数 | 解释 |
| --- | --- |
| `collection_name` | 搜索哪个 collection |
| `anns_field` | 搜索哪个向量字段 |
| `data` | 查询向量列表，可以一次传多个 query vector |
| `limit` | TopK |
| `filter` | 标量过滤表达式 |
| `output_fields` | 希望返回哪些字段 |
| `search_params` | 搜索参数，这里指定 metric |

最重要的是这三项：

```python
anns_field="vector"
data=[query_vector]
limit=3
```

它的意思是：

```text
用 query_vector 去 collection 的 vector 字段里找最相似的 3 条记录。
```

## 20. output_fields 为什么重要

代码中返回：

```python
output_fields=[
    "doc_id",
    "chunk_id",
    "title",
    "text",
    "source",
    "category",
    "tenant_id",
    "page",
]
```

如果不写 `output_fields`，你可能只拿到 id 和 score。

在 RAG 中，你至少需要：

- `text`：给 LLM 的上下文。
- `source`：答案引用来源。
- `doc_id` / `chunk_id`：追踪文档。
- `score`：判断召回质量。

生产中不要无脑返回所有字段。

原因：

- 文本字段可能很长。
- 网络传输变慢。
- 应用层处理变慢。
- 可能泄露不该返回的 metadata。

## 22. 带过滤的向量搜索

代码：

```python
filter_expr = 'tenant_id == "team_a" and category == "rag"'
results = vector_search(
    client,
    collection_name,
    query,
    limit=5,
    filter_expr=filter_expr,
)
```

这表示：

```text
先限定：
  tenant_id 必须等于 team_a
  category 必须等于 rag

再在满足条件的数据中做向量搜索。
```

在 RAG 中这是非常重要的能力。

典型场景：

```python
filter='tenant_id == "company_a"'
filter='tenant_id == "company_a" and category == "policy"'
filter='doc_id in ["doc_1", "doc_2"]'
filter='created_at >= 1710000000'
```

不要只在应用层过滤权限。

更稳妥的方式是：

```text
应用层校验用户权限
  + Milvus filter 限定 tenant_id/doc_id/category
```

## 23. print_hits()：解释搜索结果

搜索结果中的单条 hit 通常包含：

```text
id
distance
entity
```

代码打印：

```python
rank=1 id=1 score=0.4512 tenant=team_a category=milvus title=Milvus Docker 部署
```

注意：

- 使用 `COSINE` 时，分数通常越大越相似。
- 使用 `L2` 时，距离越小越相似。
- 不同 metric 的阈值不能混用。

生产中你可能会设置：

```python
if score < threshold:
    discard_result()
```

但阈值必须通过评估数据集调出来，不要拍脑袋。

## 25. get by primary keys

```python
rows = client.get(
```

## 26. upsert：插入或更新

代码：

```python
result = client.upsert(
    collection_name=collection_name,
    data=[chunk_to_entity(updated)],
)
```

`upsert` 的含义：

```text
如果主键不存在：插入
如果主键已存在：更新/覆盖
```

脚本中更新的是：

```text
id = 6
```

原本标题：

```text
Milvus 生产化关注点
```

更新后标题：

```text
Milvus 生产化关注点（更新版）
```

RAG 中什么时候用 upsert？

- 文档重新解析。
- chunk 内容变了。
- source 信息变了。
- embedding 模型不变，只更新 metadata。

如果 embedding 模型变了，通常需要批量重新生成向量并重建索引。

## 27. delete：删除数据

代码：

```python
result = client.delete(
    collection_name=collection_name,
    filter='doc_id == "infinity-intro"',
```

这表示删除：

```text
doc_id 等于 infinity-intro 的所有 chunk
```

RAG 中删除整篇文档时，通常不是按主键一条条删，而是：

```python
filter='doc_id == "xxx"'
```

删除后脚本又执行：

```python
client.query(
    collection_name=collection_name,
    filter='doc_id == "infinity-intro"',
    output_fields=["id", "title", "doc_id"],
)
```

如果返回空列表，说明删除成功。

```python
def retrieve(
```

## 31. 替换为真实 embedding

当前：

```python
"vector": embed_text(chunk.title + "\n" + chunk.text)
```

你可以替换成真实 embedding 函数。

例如伪代码：

```python
def embed_text_with_real_model(text: str) -> list[float]:
    response = embedding_client.embeddings.create(
        model="your-embedding-model",
        input=text,
    )
    return response.data[0].embedding
```

然后改：

```python
"vector": embed_text_with_real_model(chunk.title + "\n" + chunk.text)
```

同时必须改：

```python
VECTOR_DIM = 你的模型维度
```

比如：

```python
VECTOR_DIM = 768
```

注意：

- 维度改了必须重建 collection。
- 模型改了通常要重新 embedding 全量文档。
- metric type 要和模型推荐保持一致。

## 33. 常见报错处理

### 33.1 连接不上 Milvus

报错可能类似：

```text
Cannot connect to Milvus
```

检查：

```bash
docker ps
docker logs milvus
```

确认：

- 容器运行中。
- `19530` 端口暴露。
- URI 是 `http://localhost:19530`。

### 33.2 vector dimension mismatch

原因：

```text
collection vector dim = 128
insert vector dim != 128
```

解决：

```python
print(len(vector))
```

并确保：

```python
VECTOR_DIM == len(vector)
```

### 33.3 field not found

原因：

你插入的数据字段没有在 schema 中定义。

例如 schema 中没有：

```text
author
```

但插入 dict 有：

```python
{"author": "alice"}
```

由于 demo 设置：

```python
enable_dynamic_field=False
```

这种情况会报错。

### 33.4 filter 写错

正确：

```python
filter='tenant_id == "team_a" and category == "rag"'
```

常见错误：

```python
filter="tenant_id = team_a"
```

问题：

- 等号应该是 `==`。
- 字符串值要加引号。
